## Session 2: Major UI Overhaul + New Features

### Part 1: Double Window Bug Fix
**User reported:** App was launching two Electron windows.
**Root cause:** The `electron:dev` script in `package.json` was `concurrently "vite" "wait-on http://localhost:5173 && electron ."` — but `vite-plugin-electron` already launches Electron automatically, so `electron .` created a duplicate.
**Fix:** Changed script to just `"vite"`. Killed 12 leftover Electron processes. Confirmed single window launch.

### Part 2: Major UI Overhaul — User's Request
User asked for several changes in one message:

1. **Move platform pills to a sidebar** — "Instead of being across the top as pill-shaped buttons, I'd like them in their own section on the left or right side for a cleaner look"
2. **Add paste button** — Auto-paste text into whatever window had focus before VoiceType (minimize, Ctrl+V, restore)

4. **Global rules** — Cross-platform text processing rules like "never use M-dashes"
5. **Modify/Chat button** — User was asked if this should modify output text or prompt templates. Answer: **"Both"**
6. **Local ticket system** — "I would also like to add a ticketing system that I can click to add future change recommendations"

#### Planning Phase
Entered plan mode. Three parallel explore agents examined the codebase. Designed 6-phase implementation plan covering all features. User approved the plan.

#### Implementation Phases
Built everything across 6 phases:
- **Phase 1:** Foundation — types, store, LLM service, Electron IPC (global rules, paste-to-external, tickets)
- **Phase 2:** Global Rules UI — textarea + save in PromptsView
- **Phase 3:** Action Bar — Copy/Paste/Record/Clear/Redo/Modify near the mic button
- **Phase 4:** Platform Sidebar — vertical sidebar with text previews, collapsible
- **Phase 5:** Ticket System — full CRUD, status toggles, filters, priority badges
- **Phase 6:** Chat/Modify Panel — AI chat for text refinement + prompt improvement

#### Key Design Decisions
- **On-demand processing** (later changed to simultaneous) — originally only processed active platform
- **Per-platform edit preservation** — switching platforms saves editedText back to cache
- **Chat is session-only** — messages don't persist to disk
- **Paste uses PowerShell SendKeys** — zero-dependency Windows solution
- **Global rules in PromptsView** — belongs with platform prompts, not general settings
- **`<modified>` tag pattern** — AI wraps suggestions in tags, "Apply" button extracts and applies

### Part 3: Four Additional Pending-Task Requests
After seeing the UI overhaul working, user asked for 4 more changes:

#### 1. Process all platforms simultaneously
> "I'd like the little previews on the left to load up when all at one time at the beginning. When it processes it, I'd like to see them all over there."

**Implementation:** After transcription completes, fire off parallel `processText()` calls for all enabled platforms (not just the active one). Each resolves independently and updates the sidebar preview. Used `.then()` instead of `await` so they don't block each other. Raw/no-prompt platforms get the transcription directly.

**Files changed:** `src/components/MainView.tsx`

**Files changed:** `src/components/TicketsView.tsx`

#### 3. Default filler word removal global rule
> "I want all the prompts to remove words like 'um' and dead air"

**Implementation:** Changed default `globalRules` in `electron/main.ts` to: "Always remove filler words and verbal hesitations (um, uh, ah, er, like, you know, I mean, sort of, kind of). Remove dead air artifacts and false starts." Also migrates existing installations with empty globalRules.

**Files changed:** `electron/main.ts`

#### 4. Microphone volume meter
> "Could we add a microphone slider, or some way to adjust its sensitivity? It would be cool if I could see the bars as I'm testing it to see how much it's picking up."

**Implementation:**

- `getAnalyserNode()` getter for external access
- `startVisualization()` / `stopVisualization()` for browser speech mode (needs separate stream)
- MainView polls at ~30fps via `requestAnimationFrame` with throttling
- 24 frequency bars rendered with color coding (green < 0.35, amber < 0.65, red above)
- Smooth transitions with 50ms ease-out

**Files changed:** `src/services/speechService.ts`, `src/components/MainView.tsx`

## Session 3: HotKeys Overlay + Learning Mode

This session continued from where Session 2 left off. Two ticket items were implemented:

### 5. [HIGH] HotKeys — Global Hotkey Overlay
**From ticket system:** User had a HIGH priority ticket requesting global hotkeys.

**What was built:**
- **Global hotkey:** Ctrl+Shift+Space registered via Electron's `globalShortcut` API
- **Overlay window:** New 380x220 always-on-top BrowserWindow created in `electron/main.ts`
  - Frameless or minimal frame
  - Always on top of other windows
  - Contains Record, Copy, and Paste buttons
- **OverlayView.tsx:** New React component with minimal, compact UI designed for the small overlay
- **Settings integration:** Hotkey configuration added to SettingsView so users can see/change the shortcut
- **IPC bridge:** Overlay-related methods added to `electron/preload.ts`

**Design decisions:**
- Overlay is a separate BrowserWindow, not a panel in the main window — allows true always-on-top behavior
- 380x220 is compact enough to sit in a corner without obstructing work
- Escape key dismisses the overlay
- The overlay communicates with the main window's recording/clipboard services via IPC

**Files created:** `src/components/OverlayView.tsx`
**Files modified:** `electron/main.ts`, `electron/preload.ts`, `src/types/index.ts`, `src/components/SettingsView.tsx`

### 6. [MEDIUM] Learning Mode — Auto-Correction Detection
**From ticket system:** User had a MEDIUM priority ticket requesting learning mode improvements.

**What was built:**
- **Learning Mode toggle:** Added to SettingsView as an on/off switch
- **Auto-detection logic:** When Learning Mode is enabled, the app watches for text edits. When the user edits transcribed text and then copies it, the system compares the original transcription against the edited version to detect word-level corrections
- **Auto-dictionary addition:** Detected corrections are automatically added to the dictionary (e.g., if "definately" is always corrected to "definitely", it gets added)
- **Status banner in LearningView:** Shows whether Learning Mode is currently active or inactive, with a description of what it does
- **State management:** Learning mode state added to Zustand store

**Design decisions:**
- Learning mode is opt-in (toggle in Settings) — not everyone wants auto-corrections
- Banner in LearningView gives clear visibility into whether it's active
- Corrections need to be consistent (same correction multiple times) before auto-adding — prevents one-off edits from polluting the dictionary

**Files modified:** `src/components/LearningView.tsx`, `src/components/SettingsView.tsx`, `src/stores/appStore.ts`, `src/types/index.ts`

## Problems Encountered and Solutions
1. **Double window on launch** — Fixed by simplifying electron:dev script (vite-plugin-electron handles Electron startup)
2. **Recording timer drift** — Fixed by using Zustand `getState()` instead of React state callback
3. **Parallel processing** — Used `.then()` fire-and-forget pattern instead of `await` to avoid sequential blocking
4. **Browser speech visualization** — Needed a separate `startVisualization()` method because browser Web Speech API doesn't expose its audio stream
