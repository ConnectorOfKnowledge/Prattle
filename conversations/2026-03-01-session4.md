## The Bug: Blank Overlay Window

### Symptom
After implementing changes, pressing Ctrl+Shift+Space opened a completely blank white window. React did not mount at all — `<div id="root"></div>` was empty.

### User Messages
- "you'll need to restart it when you're done with changes. the pop out window is stuck after I closed the app"
- "make sure there's still an X on the overlay to manually close it without closing the whole app please" (it was already there)
- "restart again please"
- "The hotkey is only bringing up a blank window"
- Sent a screenshot showing the blank overlay with DevTools open, Elements tab showing empty root div, 4 console errors
- "no change. restart it please. that white window is still hanging there"
- "I've force closed them. try opening it again please"
- "The white window has not gone away yet. didn't even go while you restarted"

### Debugging Steps
1. Checked dev server terminal output — clean builds, no errors
2. Verified `HiChevronUpDown` icon exists in react-icons/hi2 — confirmed
3. Added DevTools to overlay window in dev mode
4. Added `did-fail-load` listener on overlay webContents
5. Changed hardcoded `http://localhost:5173` to `process.env.VITE_DEV_SERVER_URL`
6. Added error handling with visible error/loading states in OverlayView
7. User screenshot revealed empty `<div id="root">` — React not mounting
8. Identified possible hooks violation: `useEffect` after conditional `return` in `App.tsx`
9. Moved overlay detection to `main.tsx` so OverlayView renders as separate component tree
10. Fix wasn't verified — stale BrowserWindow from previous Electron session persisted on screen
11. Multiple attempts to kill Electron processes via PowerShell/taskkill
12. Stale window refused to close even after restart

### Root Cause
**Uncertain.** Two theories:
1. **React hooks violation in App.tsx:** The early return `if (isOverlay) return <OverlayView />` before `useEffect` technically violates Rules of Hooks. But `isOverlay` is a module-level constant, so hook count is consistent per-render. This code WAS working before the session.
2. **State management issue in new OverlayView code:** The `activePlatform` useState at line 200 was after the `handleToggle` useCallback at line 93 that referenced it in its dependency array. Combined with new effects, loading states, and the platform change handler, something may have caused a render cycle issue.

### Resolution
After extensive debugging failed to resolve the issue, the user requested: **"I want to go back to that version before making any further changes."**

All 8 modified files were reverted to their Session 3 state. The app was restarted and builds clean.
